# 臺北市土地開發法定結論系統
## README / Codex 專案理解文件（V1 / V2）

版本：v0.1  
適用城市：臺北市  
文件用途：提供 Codex / 開發團隊理解產品定位、法規判斷邏輯、MVP 範圍、模組拆分、優先級與輸出格式。

---

## 1. 專案定位

本專案目標是建立一套可針對「臺北市特定基地」進行**土地開發法規自動判讀**的系統。

系統名稱可使用：

**臺北市土地開發法定結論系統**

但必須明確區分：

- **V1 = 法規自動初判 / MVP**
- **V2 = 量體消化 + 停車配置 + 審議風險評估**

### 1.1 系統不是什麼

V1 不是主管機關，不可直接取代：

- 正式土地使用分區證明
- 建築線指示 / 指定之正式行政認定
- 畸零地調處結果
- 都審委員會審議結果
- 容積獎勵核給結果
- 容積移轉取得結果
- 建造執照核發結果

### 1.2 系統可以做什麼

V1 可以在輸入地號 / 地址後，快速產出：

1. 基地法規初判
2. 是否存在一票否決事項
3. 基準容積 / 建蔽 / 停車數量初算
4. 特殊疊圖風險（都審、山坡地、文資、環評）
5. 需人工覆核事項
6. 法規依據與查詢來源
7. 可做 / 高風險 / 不建議 / 需人工覆核 的結論分級

## 2. 法規判斷總原則

Codex 必須依以下層級判定：

1. **中央法律 / 中央法規命令**
2. **臺北市自治條例 / 自治規則**
3. **都市計畫書圖 / 細部計畫 / 其他規定**
4. **官方圖資與查詢系統**
5. **審議裁量 / 行政認定 / 人工覆核**

### 2.1 基本適用邏輯

- 上位法優先於下位法
- 特別法優先於普通法
- 地方規定如有加嚴，取較嚴者
- 法規明文可直接判定者，才可列為自動結論
- 涉行政裁量或審議者，一律標記為 `REVIEW_REQUIRED`

### 2.2 停車的規則樹（系統必做）

停車數量不可只套一條公式，必須依優先序判斷：

1. 先看都市計畫是否另有停車規定
2. 都市計畫未規定時，依《臺北市土地使用分區管制自治條例》第 86-1 條
3. 其餘未規定者，再回《建築技術規則》處理
4. V2 再加入車道、坡道、尺寸、集中留設與地下室效率

## 5. 產品輸入 / 輸出定義

## 5.1 最低輸入欄位

V1 最低必填：

- 地址或地號
- 基地面積
- 預計用途（住宅 / 辦公 / 商業 / 旅館 / 廠辦）

可選填：

- 是否單一地主
- 是否可整合鄰地
- 是否已有建物
- 是否已有建照 / 使照
- 預計制度（一般開發 / 都更 / 危老）

## 5.2 系統輸出

V1 必須輸出以下欄位：

- `project_id`
- `site_identity`
- `zoning_result`
- `use_result`
- `road_frontage_result`
- `building_line_result`
- `odd_lot_result`
- `far_result`
- `coverage_result`
- `parking_result`
- `overlay_risks`
- `manual_review_items`
- `final_status`
- `legal_basis`
- `source_evidence`
- `generated_at`
- `rule_version`

### 5.3 結論等級

系統統一使用以下四種狀態：

- `AUTO_PASS`：規則明確，可直接通過
- `AUTO_FAIL`：規則明確，不符合
- `REVIEW_REQUIRED`：需人工覆核
- `HIGH_RISK`：可初判，但存在重大風險

### 5.4 對外可讀結論文字

- 可初步開發
- 高風險，建議先人工複核
- 不建議取得
- 需補件 / 需人工判定

## 8. V1 明確不做的事

以下項目即使能初判，也不能在 V1 做「法定保證式結論」：

1. 都審一定會過
2. 建築線已正式合法
3. 畸零地已可完成調處
4. 容積獎勵一定能拿到多少
5. 容積移轉一定買得到
6. 建造執照一定可核發
7. 文資影響已完全排除
8. 環評一定不需要

V1 的角色是：

**把可機械判斷的部分做完，把不能機械判斷的部分明確標記出來。**

## 9. V1 規則引擎主流程

Codex 需依序執行以下流程：

### Step 1：基地正規化

輸入地址 / 地號後：
- 轉成標準地號
- 建立唯一基地識別碼
- 串接圖資座標

### Step 2：都市計畫 / 分區判定

檢查：
- 是否位於都市計畫範圍內
- 使用分區
- 其他規定
- 是否有特殊街廓 / 細部計畫附帶限制

輸出：
- 分區名稱
- 允許用途初判
- 若用途不符，直接 `AUTO_FAIL`

### Step 3：道路 / 建築線前置判定

檢查：
- 是否臨接計畫道路
- 是否屬免申請指示建築線地區
- 是否需指示建築線
- 是否需指定建築線

輸出：
- `AUTO_PASS` / `REVIEW_REQUIRED` / `AUTO_FAIL`

### Step 4：畸零地初判

檢查：
- 寬度是否達標
- 深度是否達標
- 臨建築線寬度是否達 4.8 公尺

輸出：
- 非畸零
- 疑似畸零
- 需調處 / 合併使用

### Step 5：容積 / 建蔽

檢查：
- 法定容積
- 建蔽率
- 是否有書圖加嚴

輸出：
- `base_far`
- `base_bcr`

### Step 6：停車數量初算

優先序：
- 都市計畫規定
- 土管自治條例第 86-1 條
- 建築技術規則 fallback

輸出：
- 法定汽車位數
- 法定機車位數（如適用）
- 停車計算依據

### Step 7：特殊疊圖風險

檢查：
- 都審地區
- 山坡地管制區
- 文資範圍
- 環評門檻可能性

輸出：
- `overlay_risks[]`

### Step 8：結論整合

整合所有規則：
- 若存在一票否決 → `AUTO_FAIL`
- 若無法直接判定 → `REVIEW_REQUIRED`
- 若可做但有重大風險 → `HIGH_RISK`
- 若條件明確且皆符合 → `AUTO_PASS`

## 14. 人工覆核規則

只要符合下列任一情況，系統不得強行輸出最終肯定結論：

1. 建築線需正式行政指示 / 指定
2. 畸零地可能涉及調處或鄰地整合
3. 細部計畫 / 其他規定無法完整結構化解析
4. 文資範圍邊界需正式認定
5. 都審雖適用但程序與風險需人工判讀
6. 環評門檻涉及複合型開發內容
7. 既有建物合法性查詢結果不完整

這些情況一律回傳：

`REVIEW_REQUIRED`

## 15. 對外文案建議

雖然產品名稱可叫「法定結論系統」，但 V1 對外說法應控制為：

## 18. 最終結論

### 18.1 關於你原本的 19 點

結論如下：

- 作為 **V1 的主骨架：夠**
- 作為 **V2 的完整開發評估：不夠**

### 18.2 正確產品策略

最佳產品路徑應為：

- **V1：臺北市土地開發法規自動初判系統（MVP）**
- **V2：臺北市土地開發高階評估系統**

### 18.3 MVP 成功標準

當使用者輸入一塊臺北市土地後，系統能在短時間內穩定輸出：

1. 是否有明確法規否決事項
2. 分區與用途是否相符
3. 是否疑似畸零 / 建築線風險
4. 基準容積 / 建蔽 / 停車數量
5. 是否落入都審 / 山坡地 / 文資 / 環評等特殊管制
6. 哪些項目一定要交給人工覆核

若上述 6 點能完成，即視為 **V1 MVP 成功**。
